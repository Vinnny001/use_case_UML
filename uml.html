Question 1 — <<include>> vs <<extend>> Relationships

Use a library system to illustrate the difference

Both <<include>> and <<extend>> are dependency stereotypes in UML use case diagrams, but they differ in whether the linked behaviour is mandatory or conditional.

<<include>>

The base use case always calls the included use case — the behaviour is mandatory and reusable. The arrow points from the base to the included case.

Library example: Borrow Book <<include>> Verify Membership — you can never borrow without checking membership.

<<extend>>

The extending use case optionally adds behaviour to the base under a specific condition. The arrow points from the extension to the base.

Library example: Return Book is extended by Charge Fine — a fine is only raised when the book is overdue.

Question 3 — UML and Use Case Diagram Notation

UML stands for Unified Modeling Language — a standardised visual language for specifying, visualising, constructing, and documenting software systems.

1. ActorA stick figure representing a person or external system that interacts with the system from outside its boundary.
2. Use caseAn oval/ellipse labelled with the name of a system function or user goal.
3. System boundaryA rectangle (often dashed) enclosing all use cases belonging to the system, labelled with the system name.
4. AssociationA solid line connecting an actor to a use case, showing that the actor participates in that use case.
5. «include»A dashed arrow from a base use case to the included use case. The base always calls the included behaviour.
6. «extend»A dashed arrow from the extending use case to the base. The extension is conditional/optional.
7. GeneralisationA solid line with a hollow triangle arrowhead. A child actor/use case inherits from the parent.
8. Note/CommentA dog-eared rectangle used to annotate elements with conditions, constraints, or explanatory text.

Question 5 — Activity Models in OO Modelling

An activity model (activity diagram) is a UML behavioural diagram that depicts the flow of activities — sequential and parallel — within a system or use case. It is similar to a flowchart but supports concurrent paths (fork/join bars), swim lanes (showing which actor does what), and decision points.

Why it is useful in OO modelling

  1. Workflow clarity — shows the order of operations fulfilling a use case, bridging the gap between use case diagrams (what) and sequence diagrams (how).
  2. Parallel processing — fork and join nodes model concurrency that is hard to show in a sequence diagram.
  3. Swim lanes — partition responsibilities by actor or subsystem, making it clear who performs each activity.
  4. Input to class design — activities reveal the operations that classes must expose.
  5. Validation tool — stakeholders can walk through the diagram to confirm the modelled behaviour matches real-world expectations before any code is written.
Activity diagrams are most useful during requirements analysis and high-level design, complementing use case and class diagrams.

Question 7 — Use Case, Actor, and Importance of Use Case Diagrams

A use case describes a single goal that an actor can accomplish using the system — a complete sequence of interactions producing a result of value (e.g., "Purchase flight ticket").

An actor is any entity — person, organisation, or external system — that interacts with the system from outside. Actors are not part of the system; they trigger use cases or receive results. A primary actor initiates the use case (e.g., Customer); a secondary actor supports it (e.g., Payment Gateway).

Why use case diagrams are important

  • Express requirements in plain, stakeholder-friendly language
  • Define the system boundary — what is in scope versus out of scope
  • Serve as a contract between developers and clients before design begins
  • Drive test-case design: one test scenario per use case path
  • Identify the actors and thus the user groups whose needs must be satisfied
Stage of use: Use case diagrams are most valuable during the requirements analysis phase — the earliest stage. They capture what the system must do before any design decisions about how are made. They feed directly into class, sequence, and activity diagrams in subsequent phases.

Question 10 — Why Objects Are the Key Concept in OO Development

Objects are the key concept because they mirror the way we naturally understand the real world — as distinct things with state (what they know) and behaviour (what they can do). This correspondence reduces the conceptual gap between the problem and the software solution.

  • Encapsulation — each object bundles data and methods together, reducing coupling.
  • Reusability — a well-designed class can be reused across multiple projects.
  • Modularity — a system of cooperating objects is naturally modular; changing one object's implementation does not break others if the interface is preserved.
  • Extensibility — inheritance lets new objects extend existing ones without rewriting them.
  • Naturalness — mapping real-world entities (Customer, Flight, Book) directly to objects makes code self-documenting and intuitive.

Question 12 — OO Analysis vs OO Design

Object-Oriented Analysis (OOA) focuses on understanding the problem domain — identifying what the system must do, the real-world objects involved, and their relationships. The output is a conceptual model independent of any implementation technology. Key questions: What objects exist? What do they know? What can they do?

Object-Oriented Design (OOD) focuses on planning the solution — deciding how the system will be built. It adds technical decisions: design patterns, class-to-database mappings, interface design, memory management, and concurrency. Key questions: How will objects collaborate? What classes and interfaces are needed?

The fundamental difference: analysis is problem-space thinking (technology-agnostic); design is solution-space thinking (technology-specific).

Question 13 — Four Reasons for Building Models

  1. Communication — models give developers, stakeholders, and clients a shared visual language. A class diagram communicates structure far more efficiently than prose.
  2. Complexity management — software systems are too complex to hold in one person's head. Models let you view the system at different levels of abstraction.
  3. Validation before coding — errors caught in a model cost a fraction of errors caught in code. A missing return message in a sequence diagram prevents a bug in production.
  4. Documentation and maintenance — models serve as living documentation. Diagrams explain design intent that code alone rarely conveys.

Question 14 — Key OO Distinctions

a. Encapsulation vs Data Abstraction

Encapsulation is the mechanism of bundling data (fields) and the methods that operate on that data into a single unit (a class) and restricting direct external access via access modifiers (private, protected). It is about protection and bundling.

Data abstraction is the principle of exposing only the essential characteristics of an object while hiding unnecessary implementation detail, achieved through interfaces and abstract classes. It is about simplification and hiding complexity. In short: encapsulation hides data; abstraction hides implementation.

b. Class vs Object

A class is a blueprint or template — it defines the attributes and methods that its instances will have. It exists in code but not in memory at runtime (until instantiated). An object is a specific instance of a class — a concrete entity with its own attribute values, occupying memory at runtime. For example, Flight is the class; KQ101 departing on 25 May 2026 is an object.

c. Method Overloading vs Method Overriding

Method overloading (compile-time polymorphism) — multiple methods with the same name in the same class but with different parameter lists. The compiler resolves which method to call based on arguments supplied.

Method overriding (runtime polymorphism) — a subclass provides its own implementation of a method already defined in the superclass, with the same name and signature. The decision of which method runs is made at runtime based on the actual object type.

Question 15 — Natural Language Approach to Identifying Objects

The natural language (noun-phrase) approach involves the following steps:

  1. Gather a textual description of the problem domain — a problem statement or requirements document in plain English.
  2. Identify all nouns and noun phrases — underline every noun. These are candidate objects or attributes.
  3. Identify verbs — these become candidate operations or methods.
  4. Apply filters to remove: redundant nouns (synonyms), irrelevant nouns (outside system scope), attributes masquerading as nouns, implementation constructs (database, file), and roles.
  5. What remains — nouns that are clearly distinct, relevant to the domain, and have both data and behaviour — become the object classes.
Strengths: Intuitive, requires no special notation, produces results stakeholders can review.
Weaknesses: English is ambiguous — different analysts may identify different objects. Best used as a starting point rather than a definitive method.

Question 17 — Two Types of Association Relationships

1. Unidirectional Association

One class knows about and uses the other, but not vice versa. Navigation goes in one direction only, shown by an arrowhead on one end.

Example: A Customer places a Reservation. Customer holds a reference to Reservation to check booking status, but a Reservation does not need a reference back to the Customer. Arrow: Customer → Reservation.

Limits coupling — the Reservation class remains reusable without being tied to any particular Customer implementation.

2. Bidirectional Association

Both classes know about and can navigate to each other. Represented by a plain line with no arrowhead (or arrowheads on both ends).

Example: A Doctor treats a Patient, and a Patient is treated by a Doctor. Both classes hold references to each other because the system must answer: "Which patients does this doctor treat?" and "Which doctor is treating this patient?"

More common in domain models but increases coupling — reduce to unidirectional wherever only one direction of navigation is actually needed.
